# 第6章 链接

## 1. 本章概述

链接（Linking）是将多个独立编译的代码模块（目标文件）合并成一个可执行文件的过程。本章讲解链接的工作原理，包括目标文件格式、符号解析规则、重定位机制以及静态链接与动态链接的区别。

## 2. 知识点讲解

### 2.1 什么是链接

**通俗定义：**
链接就像搭积木——你把程序拆成多个源文件分别写好（像分开的积木块），链接器负责把这些积木块拼成一个完整的可执行程序。它要解决的问题是：模块 A 调用了模块 B 中的函数，但编译 A 时还不知道 B 中函数的地址，链接器就得把这些"悬空"的引用填上正确的地址。

**标准公式（概念无公式，用流程描述）：**

```
源文件1.c ──编译──→ 可重定位目标文件1.o ──┐
源文件2.c ──编译──→ 可重定位目标文件2.o ──┤
源文件3.c ──编译──→ 可重定位目标文件3.o ──┼──→ 链接器(ld) ──→ 可执行目标文件
库文件(libc.a) ──────────────────────────┘
```

**举例：**
当你写 `#include <stdio.h>` 并调用 `printf()` 时，你的源文件并没有 `printf` 的机器代码。编译后 `printf` 的地址是待定的，链接器会从 C 标准库中找到 `printf` 的代码，把它链接到你的程序中。

**初学易错点：**
- 链接 ≠ 编译。编译是把源文件变成机器指令（目标文件），链接是把多个目标文件和库合并成可执行文件。初学者常把"编译"当作整个过程，实际上编译只是其中一步。
- 链接发生在程序运行之前（静态链接），也可能发生在加载或运行时（动态链接）。

---

### 2.2 GCC 编译的四个步骤

**通俗定义：**
用 GCC 从 C 源文件生成可执行文件，要经过四个阶段：预处理 → 编译 → 汇编 → 链接。每个阶段做不同的工作，输出不同的中间文件。

**标准公式：**

```
源文件(.c) ──预处理──→ 预处理文件(.i) ──编译──→ 汇编文件(.s) ──汇编──→ 目标文件(.o) ──链接──→ 可执行文件
```

| 步骤 | 命令 | 输入 | 输出 | 说明 |
|:---:|:---|:---|:---|:---|
| ① 预处理 | `gcc -E` | `main.c` | `main.i` | 处理 `#include`、`#define`、条件编译等 |
| ② 编译 | `gcc -S` | `main.i` | `main.s` | 将 C 代码翻译成汇编代码 |
| ③ 汇编 | `gcc -c` | `main.s` | `main.o` | 将汇编代码翻译成机器指令（二进制目标文件） |
| ④ 链接 | `gcc`（无特殊选项） | `main.o` + 库 | `a.out` | 合并目标文件和库，生成可执行文件 |

**举例：**
```bash
# 完整四个步骤的手动执行
gcc -E hello.c -o hello.i      # 预处理
gcc -S hello.i -o hello.s      # 编译
gcc -c hello.s -o hello.o      # 汇编
gcc hello.o -o hello           # 链接
# 等价于一步完成
gcc hello.c -o hello
```

**初学易错点：**
- 误区："第二步编译"的结果是二进制目标文件——**错误**。第二步（编译）的结果是汇编文件（`.s`，文本），第三步（汇编）才产生二进制目标文件（`.o`）。
- 预处理阶段处理的是 `#include`、`#define` 等以 `#` 开头的指令，不是处理 C 语法。
- 汇编阶段不是"把汇编翻译成可执行文件"，而是翻译成**可重定位目标文件**（`.o`），还需要链接才能运行。

---

### 2.3 目标文件与 ELF 格式

**通俗定义：**
目标文件是编译/汇编后的中间产物，它包含机器指令和数据，但其中的地址还没有最终确定（需要链接器来填）。Linux 上使用的目标文件格式叫 ELF（Executable and Linkable Format，可执行可链接格式）。ELF 文件像一本带目录的书，分成多个节（section），每个节存放一类信息。

**ELF 主要节（Section）：**

| 节名 | 中文名 | 存放内容 |
|:---|:---|:---|
| `.text` | 代码节 | 机器指令（程序的可执行代码） |
| `.data` | 数据节 | **已初始化**的全局变量和静态变量 |
| `.bss` | 未初始化数据节 | **未初始化**的全局变量和静态变量（在文件中仅占位，不占磁盘空间） |
| `.rodata` | 只读数据节 | 只读数据（如字符串常量、`const` 修饰的变量） |
| `.symtab` | 符号表 | 符号信息（函数名、全局变量名等） |
| `.rel.text` | 代码重定位节 | `.text` 节中需要修改的地址位置列表 |
| `.rel.data` | 数据重定位节 | `.data` 节中需要修改的地址位置列表 |

**三种目标文件类型：**

| 类型 | 扩展名 | 说明 |
|:---|:---|:---|
| 可重定位目标文件 | `.o` | 包含代码和数据，可与其他目标文件链接 |
| 可执行目标文件 | 无（或 `a.out`） | 可直接加载到内存执行 |
| 共享目标文件 | `.so`（动态链接库） | 可被动态链接或加载 |

**标准公式（ELF 文件结构示意）：**

```
ELF头 ──→ 包含文件类型、机器架构、入口点地址、节表偏移等信息
   │
   ├── .text节    （机器指令）
   ├── .rodata节  （只读数据）
   ├── .data节     （已初始化全局/静态变量）
   ├── .bss节      （未初始化全局/静态变量，不占磁盘空间）
   ├── .symtab节   （符号表）
   ├── .rel.text节 （.text的重定位信息）
   ├── .rel.data节 （.data的重定位信息）
   └── 节头部表    （描述各节的位置和大小）
```

**举例：**
```c
// 假设有这样一个 C 文件
int global_init = 42;                 // → .data节（已初始化全局变量）
int global_uninit;                     // → .bss节（未初始化全局变量）
static int static_init = 10;          // → .data节（已初始化静态变量）
static int static_uninit;             // → .bss节（未初始化静态变量）
const int const_val = 100;            // → .rodata节（只读数据）
char *str = "hello";                  // "hello" → .rodata节，str指针 → .data节
void func() { /* ... */ }             // → .text节（函数代码）
```

**初学易错点：**
- `.data` 节**只包含已初始化的**全局变量和静态变量。未初始化的全局变量在 `.bss` 节中。题目常考"所有变量的初始值都在.data节"——这是错的。
- `.bss` 节在目标文件中不占用实际磁盘空间，仅在节表中记录大小，程序加载时由操作系统分配并初始化为 0。
- `.rodata` 中的"r"代表"read-only"（只读），不是"relocation"。
- 局部变量（函数内定义的变量）不存储在 ELF 的节中，它们在运行时分配在栈上。

### 2.4 符号与符号解析

**通俗定义：**
符号（Symbol）就是程序中定义的函数名、全局变量名、静态变量名。当模块 A 引用模块 B 中定义的符号时，链接器需要找到这个符号的定义，这就是**符号解析**。

**符号分类：**

| 类别 | 英文 | 定义方式 | 特点 |
|:---|:---|:---|:---|
| **强符号** | Strong Symbol | 有初始化的全局变量、有函数体的函数 | 每个强符号在整个程序中只能定义一次 |
| **弱符号** | Weak Symbol | 未初始化的全局变量 | 可以定义多次，链接器会选择其中一个 |
| **局部符号** | Local Symbol | `static` 修饰的变量或函数 | 只在定义它的模块内可见 |

> 注意：函数内的局部变量（非 `static`）不是符号，它们分配在栈上，不参与链接。

**强弱符号规则（Unix 链接器规则）：**

```
规则 1：强符号不能重复定义
  ── 两个模块定义了同名的强符号 → 链接错误（重定义）

规则 2：一个强符号 + 多个弱符号 → 选择强符号
  ── 链接器用强符号的定义，忽略弱符号

规则 3：多个弱符号 → 任选一个（通常选占用空间最大的）
  ── 不确定行为，可能导致隐蔽的 bug
```

**举例：**

```c
/* m1.c */
int x = 7;          // 强符号（有初始化）
int y = 5;          // 强符号（有初始化）
void p1() { /*...*/ }  // 强符号（有函数体）
static void p2() { x = 3; }  // 局部符号，只在 m1.c 可见

/* m2.c */
int x;              // 弱符号（未初始化）
void main() {
    p2();           // 调用的是 m1.c 中的 p2（静态函数是局部的）
    p1();
}
```

在此例中：
- 强符号：`x`（m1.c，已初始化）、`y`、`p1`、`main`
- 弱符号：`x`（m2.c，未初始化）
- 链接时，两个 `x` 发生冲突：`x` 在 m1.c 中是强符号（有初始化 `=7`），在 m2.c 中是弱符号（未初始化），根据规则 2，链接器选择强符号的定义，即 m1.c 中初始化为 7 的 `x`。

**初学易错点：**
- 不要把"弱符号"和"extern 声明"搞混。`extern int result;` 只是声明，不是定义，它不产生符号。所以问"当前模块中有哪些弱符号"时，`extern` 变量不算。
- `static` 变量和函数是**局部符号**，只在当前模块可见，不参与跨模块的符号解析。链接时两个文件各有同名 `static` 函数不会冲突。
- 弱符号规则可能导致极其隐蔽的 bug：比如一个模块定义了 `int x = 7`（强），另一个模块定义了 `int x`（弱），链接后两个模块共享同一个 `x`，值可能被意外修改。

### 2.5 重定位

**通俗定义：**
重定位（Relocation）是链接器的核心工作。编译时，编译器不知道变量和函数的最终地址（因为模块还没合并），所以先用临时占位。链接器把所有目标文件合并后，知道了每个符号的最终地址，再回去修改这些占位，填入正确的地址。

**标准公式：**

```
重定位的两步：
  ① 节合并与地址分配 —— 将所有目标文件的相同节合并，分配虚拟地址
  ② 符号重定位 —— 根据重定位条目(.rel.text, .rel.data)，修改引用处的地址值

重定位条目结构：
  offset: 需要修改的位置（距节起始的偏移）
  symbol: 引用的目标符号
  type:   重定位类型（如何计算新地址）
```

**举例：**

```
假设文件 a.o 和 b.o 链接：
  a.o 调用 b.o 中的函数 sum()
  
  ① 链接前 a.o 中 call sum 的地址是占位（如 0x00000000）
  ② 链接器确定最终的.text布局：
     a.o 的 .text → 地址 0x4004e0 ~ 0x4004f8
     b.o 的 .text → 地址 0x4004f8 ~ 0x400510
  ③ 找到 .rel.text 中 call sum 的重定位条目
  ④ 将 call 指令中的地址改为 sum 的最终地址 0x4004f8
```

**初学易错点：**
- 重定位不是在源文件中修改，而是在目标文件（`.o`）中修改机器指令的操作数字段。
- 不是所有引用都需要重定位——引用同一模块内的符号（特别是使用相对寻址的）可能不需要。

---

### 2.6 静态链接与动态链接

**通俗定义：**
- **静态链接**：在编译时就把所有需要的库函数代码复制到可执行文件中。好处是程序独立运行，坏处是文件大、多个程序重复包含相同库代码。
- **动态链接**：可执行文件中只记录需要哪些库的引用，运行时才加载库。好处是节省磁盘和内存、库更新无需重新编译程序。

**标准公式：**

```
静态链接：
  gcc -static main.o -o main
  → 可执行文件中包含 libc 等库的全部代码

动态链接（默认）：
  gcc main.o -o main
  → 可执行文件记录了对 libc.so 的依赖，运行时动态加载

动态链接的好处：
  · 多个程序共享库代码（内存只加载一份）
  · 库更新只需替换 .so 文件，无需重新编译程序
  · 可执行文件更小
```

**举例：**
```
$ gcc -static hello.c -o hello_static   # 静态链接，文件大小 ~800KB
$ gcc hello.c -o hello_dynamic           # 动态链接，文件大小 ~16KB
$ ls -lh hello_*
-rwxr-xr-x 1 user user 800K  hello_static
-rwxr-xr-x 1 user user  16K  hello_dynamic
```

**初学易错点：**
- 动态链接的可执行文件**不包含**库函数代码，只包含引用（如函数名和库名）。运行到 `printf` 调用时，操作系统会去加载 `libc.so` 找到 `printf` 的地址。
- 动态链接也不是完美的——如果系统缺少对应版本的 `.so` 文件，程序会报错无法运行（所谓的"DLL地狱"）。

---

### 2.7 动态链接的运行时加载（dlopen 系列）

**通俗定义：**
除了程序启动时自动加载动态库，程序运行时也可以手动加载和卸载动态库，使用 `dlopen()`、`dlsym()`、`dlclose()` 等函数。这为插件系统提供了支持。

**核心函数：**

| 函数 | 功能 |
|:---|:---|
| `dlopen("libfoo.so", flag)` | 打开（加载）动态库，返回句柄 |
| `dlsym(handle, "func_name")` | 在已加载的库中查找符号地址 |
| `dlclose(handle)` | 关闭（卸载）动态库 |

**举例：**
```c
// 运行时加载 libvector.so，查找 addvec 函数并调用
void *handle = dlopen("./libvector.so", RTLD_LAZY);
void (*addvec)(int*, int*, int*, int) = dlsym(handle, "addvec");
(*addvec)(x, y, z, 2);  // 通过函数指针调用
```

**初学易错点：**
- 使用 `dlopen` 加载的库，其中的符号（如 `addvec`）不是普通的全局符号。当题目问"以下代码中有哪些符号"时，`handle` 是变量符号，`addvec` 也是变量符号（它是函数指针变量），但 `dlopen` 加载的库中的函数名并不是当前模块的符号。
- 符号的定义是在编译/链接时确定的，`dlsym` 是在运行时查找，和编译时的符号解析不同。

## 3. 典型例题

### 例题 1 （来源：[Q0085]）

**题目：**
以下是有关使用 GCC 生成 C 语言程序的可执行文件的叙述，错误的是（ ）

A. 第四步链接，将多个模块的机器语言代码链接生成可执行目标程序文件
B. 第三步汇编，将汇编语言代码汇编转换为机器指令表示的机器语言代码
C. 第一步预处理，对 `#include`、`#define` 等预处理命令进行处理
D. 第二步编译，将预处理结果编译转换为二进制目标文件

**解答：**

逐一分析每个选项：

- **选项 A**：第四步是链接，把多个目标文件和库合并成可执行文件。这里的"机器语言代码"指的就是目标文件中的机器指令。**A 正确。**

- **选项 B**：第三步是汇编，汇编器将汇编代码（`.s`，文本格式）转换为机器指令（`.o`，二进制格式）。**B 正确。**

- **选项 C**：第一步是预处理，处理 `#include`、`#define`、`#ifdef` 等以 `#` 开头的指令。**C 正确。**

- **选项 D（错误项）**：第二步是编译，将 C 代码翻译成**汇编代码**（`.s` 文件，文本格式），不是直接生成二进制目标文件。生成二进制目标文件是第三步（汇编）的工作。

**答案：D**

**解析：**
这道题考查 GCC 编译的四个步骤及其产物。常见的错误就是把"编译"和"汇编"搞混——编译产生汇编代码（文本），汇编产生目标代码（二进制）。四个步骤的输出依次是：`.i`（预处理后的 C 代码）→ `.s`（汇编代码）→ `.o`（可重定位目标文件）→ 可执行文件。

### 例题 2 （来源：[Q0086]）

**题目：**
关于链接的叙述，下列错误的是（ ）

A. 使得公共函数库可以为所有程序共享使用，有利于代码重用和提高效率
B. 使得所生成的可执行目标代码中包含了更多公共库函数代码，所占空间大
C. 使得程序员仅需要重新编译修改过的源程序模块，从而节省程序的构建时间
D. 使得程序员可以分模块开发程序，有利于提高大规模程序的开发效率

**解答：**

逐一分析每个选项：

- **选项 A**：链接允许使用共享库（`.so`），多个程序可共享同一份库代码，确实有利于代码重用。**A 正确。**

- **选项 B（错误项）**：这个说法说"链接使得可执行文件中包含更多公共库函数代码，所占空间大"——但这是**静态链接**的情况，且只是其中一种选择。链接也支持**动态链接**，动态链接时库代码不在可执行文件中。所以不能一概而论地说链接一定会让文件变大。而且"使得"一词暗示这是链接的必然结果，不符合事实。**B 错误。**

- **选项 C**：链接支持分离编译——修改一个模块只需重新编译那个模块，再链接一次即可，不用重新编译所有文件。**C 正确。**

- **选项 D**：链接的核心价值之一就是支持模块化开发，每个人负责不同模块，最后链接在一起。**D 正确。**

**答案：B**

**解析：**
本题的关键在于理解链接的两种方式。静态链接把库代码复制到可执行文件中，确实会增加文件大小；但动态链接只保存引用，可执行文件很小，运行时共享库在内存中只加载一份。链接本身并不"必然导致文件变大"。另外，选项 B 中的"使得"表述把可能的副作用说成了必然结果，这也是判断依据。

### 例题 3 （来源：[Q0087]）

**题目：**
以下关于可重定位目标文件的叙述中，错误的是（ ）

A. 在 `.rel.text` 节和 `.rel.data` 节中包含相应模块内所有可重定位信息
B. 在 `.rodata` 节中包含相应模块内的只读数据
C. 在 `.data` 节中包含相应模块内所有变量的初始值
D. 在 `.text` 节中包含相应模块内的机器指令

**解答：**

逐一分析每个选项：

- **选项 A**：`.rel.text` 存放 `.text` 节中需要重定位的位置列表，`.rel.data` 存放 `.data` 节中需要重定位的位置列表。**A 正确。**

- **选项 B**：`.rodata` 存放只读数据，如字符串常量、`const` 修饰的全局变量等。**B 正确。**

- **选项 C（错误项）**：`.data` 节只包含**已初始化**的全局变量和静态变量的初始值。未初始化的全局变量和静态变量存放在 `.bss` 节中。选项说"所有变量的初始值"——这是错误的，因为它把 `.bss` 中的变量排除在外了。**C 错误。**

- **选项 D**：`.text` 节存放程序编译后的机器指令。**D 正确。**

**答案：C**

**解析：**
本题考查 ELF 文件中各节存储的内容。最常考的知识点是：`.data` 只放**已初始化**的全局/静态变量，`.bss` 放**未初始化**的全局/静态变量。注意区分：
- `int x = 5;` → 已初始化全局变量 → `.data`
- `int x;` → 未初始化全局变量 → `.bss`
- `static int y;` → 未初始化静态变量 → `.bss`
- `const int z = 10;` → 只读数据 → `.rodata`

### 例题 4 （来源：[Q0084]）

**题目：**
观察以下代码并回答问题。

```
/* m1.c */
int x = 7;
int y = 5;
p1() {
    // do something
}
static void p2() {
    x = 3;
}
```

```c
/* m2.c */
int x;
static void p2() {
    x = 0;
}
void main() {
    p2();
    p1();
}
```

m1.c, m2.c 链接后生成可执行程序，程序运行时的 main 函数中，在 p2 调用后 p1 调用前的时刻，变量 x 中的值为（ ）

A. 5
B. 7
C. 3
D. 0

**解答：**

**第一步：分析符号属性**

先看每个模块中的符号定义：

**m1.c：**
- `int x = 7;` — 已初始化全局变量 → **强符号**
- `int y = 5;` — 已初始化全局变量 → **强符号**
- `p1()`（有函数体）— **强符号**
- `static void p2()` — 局部符号（`static`），只在 m1.c 可见

**m2.c：**
- `int x;` — 未初始化全局变量 → **弱符号**
- `static void p2()` — 局部符号（`static`），只在 m2.c 可见
- `void main()` — **强符号**

注意：两个 `p2` 都是 `static`，各有各的作用域，互不影响！

**第二步：符号解析**

全局变量 `x` 在两个文件中都有定义：
- m1.c 中 `x` 是强符号（已初始化 `= 7`）
- m2.c 中 `x` 是弱符号（未初始化）

根据强弱符号规则：强符号胜出，弱符号被忽略。所以链接后只有一个 `x`，其初始值为 7（来自 m1.c 的定义）。

**第三步：跟踪程序执行**

```
main() 开始执行：
  ① p2() 被调用
      → m2.c 中的 p2（因为 main 在 m2.c 中，调用的是本模块的 static p2）
      → 执行 x = 0;
      → 这里的 x 是全局变量 x（链接后唯一的那个 x）
      → x 的值变为 0

  ② p1() 被调用（此时为题目问的时刻）
      → 在调用 p1() 之前，x 的值是 0
```

注意：为什么 `main` 中调用的 `p2()` 是 m2.c 的版本而不是 m1.c 的？因为两个 `p2` 都是 `static` 的——`static` 函数只在定义它的文件中可见。m2.c 中的 `main` 只能看到 m2.c 中的 `p2`，所以调用的是 m2.c 的 `p2`（将 x 设为 0）。

**答案：D（0）**

**解析：**
这道题的陷阱在于两个 `static void p2()`。很多同学会忽略 `static` 的作用，以为两个 `p2` 冲突了。实际上 `static` 使得每个 `p2` 只在自己的文件中可见，它们互不干扰。而全局变量 `x` 则参与跨模块的符号解析：m1.c 中已初始化的 `x` 是强符号，m2.c 中未初始化的 `x` 是弱符号，强符号胜出。

推理流程：
1. 确定符号强弱 → 2. 符号解析确定全局 `x` 引用哪个定义 → 3. 确定 `p2` 各自的作用域 → 4. 模拟执行，跟踪 `x` 的值的变化

### 例题 5 （来源：[Q0023]）

**题目：**
三、阅读以下 C 代码，回答问题。

```c
/* m1.c */
short x = 0x1234;
int y = 0x5;
void p1() {
    x++;
}
static void p2() {
    x = x + 2;
}
```

```c
/* m2.c */
short x;
void main() {
    p2();
    printf("x1: %d\n", x);
    p1();
    printf("x2: %d\n", x);
}
```

（1）m1.c 和 m2.c 中的强符号有哪些？
（2）m1.c 和 m2.c 中的弱符号有哪些？

**解答：**

**（1）强符号的判断：**

强符号的定义：有初始化的全局变量、有函数体的函数。

| 模块 | 符号名 | 类型 | 是否有初始化/函数体 | 强弱 |
|:---|:---|:---|:---|:---:|
| m1.c | `x` | 全局变量 | 有初始化 `= 0x1234` | **强** |
| m1.c | `y` | 全局变量 | 有初始化 `= 0x5` | **强** |
| m1.c | `p1` | 函数 | 有函数体 | **强** |
| m1.c | `p2` | 函数（static） | 有函数体 | **局部符号**（不参与跨模块符号解析） |
| m2.c | `x` | 全局变量 | 无初始化（只有声明 `short x;`） | **弱** |
| m2.c | `main` | 函数 | 有函数体 | **强** |

m1.c 中的强符号：`x`、`y`、`p1`
m2.c 中的强符号：`main`

（注意：`p2` 是 `static` 函数，属于局部符号，不是强符号也不是弱符号。）

**（2）弱符号的判断：**

弱符号的定义：未初始化的全局变量。

m1.c 中的弱符号：无（所有全局变量都有初始化）
m2.c 中的弱符号：`x`

**答案：**
（1）m1.c 强符号：`x`、`y`、`p1`；m2.c 强符号：`main`
（2）m1.c 弱符号：无；m2.c 弱符号：`x`

**解析：**
这道题考查对强弱符号的区分。关键点：
1. 有初始化的全局变量 → **强符号**
2. 未初始化的全局变量 → **弱符号**
3. 有函数体的函数 → **强符号**
4. `static` 修饰的变量或函数 → **局部符号**，不参与跨模块的强弱符号规则

补充：如果继续执行这个程序：
- `p2()` 被调用：`main` 是 m2.c 的，但 `p2` 是 `static` 的，所以 `main` 中调用的 `p2()` 是谁？——**实际上 m2.c 中没有定义 p2！** m2.c 中既没有定义自己的 `p2`，也看不到 m1.c 中的 `static p2`（因为是 `static`）。所以这段代码**编译时会报错**，因为 `main` 中引用了 `p2` 但找不到定义。这个细节也值得注意。

### 自测题 1 （来源：[Q0006]）

**题目：**
哪一个选项不是下面代码中的符号？（ ）

```c
int data[100];
int main()
{
    static void * handle;
    void (* addvec)(int*, int *, int *, int);
    handle = dlopen("./libvector.so", RTLD_LAZY);
    addvec = dlsym(handle, "addvec");
    addvec(x, y, z, 2);
    return 0;
}
```

A. data
B. main
C. handle
D. addvec

**提示：**
符号分为全局符号和局部符号。全局变量 `data` 和函数 `main` 显然是符号。`handle` 是 `static` 的局部变量——它也是符号（局部符号）。`addvec` 是一个局部变量（函数指针），它不是符号。另外，`dlopen` 加载的库中的函数（`addvec` 指向的函数）也不是当前模块的符号。

> [!question]- 点击查看完整解答
> **解答：**
> 
> 逐项判断每个名字是否属于当前模块的符号：
> 
> - **A. `data`**：全局变量 `int data[100]` 是全局符号（有定义、有初始化——虽然数组未显式初始化，但作为全局变量，它分配在 `.bss` 节中，是一个符号定义）。**是符号。**
> 
> - **B. `main`**：函数定义，是全局符号。**是符号。**
> 
> - **C. `handle`**：`static void *handle` 是 `static` 修饰的局部变量。注意这里的 `static` 使得 `handle` 具有静态存储期（存储在数据段而不是栈上），它是一个**局部符号**（只在当前文件可见）。**是符号。**
> 
> - **D. `addvec`**：`void (*addvec)(...)` 是一个局部变量（函数指针），它存储在栈上，不是全局变量也不是 `static` 变量，因此**不是符号**。注意区分：`addvec`（变量本身）和 `dlsym` 返回的地址所指向的函数（在 `libvector.so` 中定义）都不是当前编译单元的符号。
> 
> **答案：D**
> 
> **解析：**
> 本题的关键是理解"符号"的定义——符号是**链接器**处理的对象，包括：全局变量、函数、`static` 变量（静态局部变量、静态全局变量、静态函数）。不包括局部变量（自动变量）。`addvec` 是函数内的局部指针变量，不是符号。而 `handle` 虽然是局部变量，但它有 `static` 修饰，具有静态存储期，属于局部符号。
> 
> 容易混淆的点：
> - `static` 局部变量 → 是符号（局部符号）
> - 普通局部变量 → 不是符号
> - `dlsym` 获取的函数地址指向的函数 → 不是当前模块的符号，是共享库中的符号

### 自测题 2 （来源：[Q0088]）

**题目：**
阅读以下代码，并回答问题。

```c
extern int result;
static int array[2] = {1,2};
int *arr = array;
void sum() {
    int i;
    for (i=0; i<2; i++)
        result += arr[i];
}
```

在上面的代码中，属于弱符号的是（ ）

A. i
B. 没有弱符号
C. arr
D. array
E. result
F. sum

**提示：**
逐一分析每个名字：`result` 是 `extern` 声明（不是定义），`array` 是已初始化的静态全局变量（强符号），`arr` 是已初始化的全局指针变量（强符号），`sum` 有函数体（强符号），`i` 是局部变量（不是符号）。

> [!question]- 点击查看完整解答
> **解答：**
> 
> 逐项分析：
> 
> - **A. `i`**：函数 `sum` 中的局部变量（自动变量），分配在栈上，**不是符号**。
> 
> - **B. 没有弱符号**：先看其他选项，如果都不符合，则选 B。
> 
> - **C. `arr`**：`int *arr = array;` —— 已初始化的全局指针变量，**强符号**。
> 
> - **D. `array`**：`static int array[2] = {1,2};` —— 已初始化的静态全局变量，**强符号**。
> 
> - **E. `result`**：`extern int result;` —— 这是一个**声明**（declaration），不是**定义**（definition）。它告诉编译器 `result` 在其他文件中定义，但当前模块并没有定义 `result`。所以 `result` 不在当前模块的符号表中，**非强非弱**。
> 
> - **F. `sum`**：有函数体，**强符号**。
> 
> 综上，当前模块中所有符号都是强符号，没有弱符号。
> 
> **答案：B**
> 
> **解析：**
> 本题的陷阱在于 `extern int result`。很多同学会认为 `extern` 声明的变量是弱符号——这是错误的。
> 
> 弱符号的定义是：**未初始化的全局变量定义**。例如 `int x;`（在文件作用域中）。
> 
> `extern int result;` 不是定义，它只是声明"在其他地方有一个叫 `result` 的变量"，并没有为 `result` 分配存储空间。链接时，如果当前模块中使用 `result`，链接器会去其他模块找 `result` 的定义。
> 
> 记忆方法：
> - `int x;`（文件作用域）→ 弱符号（未初始化，但有分配空间的意图）
> - `extern int x;`  → 不是符号定义（只是引用声明）
> - `int x = 1;`    → 强符号（已初始化）
> - `void f() {...}` → 强符号（有函数体）
> - `static int x;`  → 局部符号（文件内可见，不参与跨模块解析）
